When a Stan API request fails, the server returns a JSON body alongside the appropriate HTTP status code. Knowing how to interpret these responses — and when to retry — is essential for building a resilient integration.

## Error Response Format

All Stan API error responses share a consistent JSON shape with three fields.

```json
{
  "statusCode": 401,
  "message": "Unauthorized",
  "error": "Unauthorized"
}
```

| Field | Type | Description |
|---|---|---|
| `statusCode` | `integer` | The HTTP status code, mirrored in the response body. |
| `message` | `string` | A human-readable description of the error. |
| `error` | `string` | A short error category label, typically matching the HTTP status phrase. |

Both `message` and `error` are intended for debugging. Do not build application logic around their exact string values, as they may change between API versions.

## HTTP Status Codes

<AccordionGroup>
  <Accordion title="200 OK — Success">
    **What it means:** The request was accepted and processed successfully. The response body contains the requested data.

    **Common causes:** N/A — this is the expected response for a well-formed request.

    **What to do:** Read the response body and continue with normal application flow. No retry is needed.

    ```json
    // Example: a successful profile fetch returns user data directly
    {
      "userId": "u_abc123",
      "username": "fragging_king",
      "platform": "bgmi"
    }
    ```
  </Accordion>

  <Accordion title="400 Bad Request — Invalid Parameters">
    **What it means:** The server understood the request but rejected it because the request body, query parameters, or headers are malformed or fail validation.

    **Common causes:**
    - Missing required fields in a POST/PATCH body
    - A field value is the wrong type (for example, a string where an integer is expected)
    - A query parameter is out of the accepted range

    **What to do:** Do not retry the request with the same payload. Read the `message` field for details about which parameter failed validation, fix your request, and resend it.

    ```json
    {
      "statusCode": 400,
      "message": "gameId must be a valid string",
      "error": "Bad Request"
    }
    ```
  </Accordion>

  <Accordion title="401 Unauthorized — Authentication Failure">
    **What it means:** The request did not carry valid authentication credentials. The API could not verify the caller's identity.

    **Common causes:**
    - The `authorization` header is missing entirely
    - The Bearer token is malformed (for example, missing the `Bearer ` prefix)
    - The JWT has expired and has not been refreshed
    - The token was issued for a different environment or has been revoked

    **What to do:** Do not retry the same request. Instead, check that the `authorization` header is present and correctly formatted as `Bearer <token>`. If the token has expired, re-authenticate to obtain a fresh JWT, then repeat the original request with the new token.

    ```json
    {
      "statusCode": 401,
      "message": "Unauthorized",
      "error": "Unauthorized"
    }
    ```

    <Warning>
      A 401 on every request often indicates a systemic token issue — for example, your token refresh logic is broken or you are reading the token from the wrong storage location. Verify the full authentication flow before retrying.
    </Warning>
  </Accordion>

  <Accordion title="404 Not Found — Resource or Endpoint Missing">
    **What it means:** The server could not find the requested endpoint path or the specific resource identified in the URL.

    **Common causes:**
    - A typo in the endpoint path (for example, `/v1/profil` instead of `/v1/profile`)
    - The resource ID in the URL (such as a user ID or game account ID) does not exist
    - You are calling an endpoint that has been deprecated or removed
    - The endpoint exists but requires a different HTTP method

    **What to do:** Do not retry the request. Verify the endpoint path against the API reference documentation. If you are fetching a resource by ID, confirm the ID is valid and belongs to the authenticated user.

    ```json
    {
      "statusCode": 404,
      "message": "Cannot GET /",
      "error": "Not Found"
    }
    ```
  </Accordion>

  <Accordion title="429 Too Many Requests — Rate Limit Exceeded">
    **What it means:** Your integration has sent too many requests in a short time window and has been temporarily throttled.

    **Common causes:**
    - Sending requests in a tight loop without any delay
    - Parallelising too many concurrent requests from the same session or IP
    - Repeatedly retrying failed requests without a backoff delay

    **What to do:** Stop sending requests immediately. Wait before retrying using **exponential backoff** — see the [Retry Strategy](#retry-strategy) section below. Check the response headers for a `Retry-After` value if present.

    ```json
    {
      "statusCode": 429,
      "message": "Too Many Requests",
      "error": "Too Many Requests"
    }
    ```
  </Accordion>

  <Accordion title="500 Internal Server Error — Server-Side Failure">
    **What it means:** The Stan API encountered an unexpected condition and could not complete the request. This is not caused by anything wrong with your request.

    **Common causes:**
    - A transient infrastructure or database issue on Stan's side
    - A recently deployed change that introduced a regression
    - An upstream service that Stan depends on is temporarily unavailable

    **What to do:** Retry the request using **exponential backoff** — see the [Retry Strategy](#retry-strategy) section below. If the error persists for more than a few minutes, the issue is likely on Stan's side and not recoverable by retrying.

    ```json
    {
      "statusCode": 500,
      "message": "Internal Server Error",
      "error": "Internal Server Error"
    }
    ```
  </Accordion>
</AccordionGroup>

## Retry Strategy

Not every error warrants a retry. Follow these rules to avoid making a problem worse.

### When to retry

| Status Code | Retry? | Strategy |
|---|---|---|
| `200 OK` | No | Success — no retry needed |
| `400 Bad Request` | ❌ No | Fix your request first |
| `401 Unauthorized` | ❌ No | Re-authenticate, then resend |
| `404 Not Found` | ❌ No | Verify the URL or resource ID |
| `429 Too Many Requests` | ✅ Yes | Exponential backoff |
| `500 Internal Server Error` | ✅ Yes | Exponential backoff |

<Warning>
  Do not retry `4xx` errors (except `429`) automatically. They indicate a problem with your request that will not resolve itself on its own. Retrying blindly wastes quota and may trigger rate limits.
</Warning>

### Exponential backoff algorithm

When you receive a `429` or `5xx` response, wait before retrying. Each successive retry should wait twice as long as the previous one, with a small random jitter added to prevent multiple clients from retrying in lockstep.

```kotlin
import kotlin.math.min
import kotlin.random.Random

suspend fun retryWithBackoff(
    maxRetries: Int = 5,
    initialDelayMs: Long = 500,
    maxDelayMs: Long = 30_000,
    block: suspend () -> Response
): Response {
    var currentDelay = initialDelayMs

    repeat(maxRetries) { attempt ->
        val response = block()

        if (response.code != 429 && response.code !in 500..599) {
            return response // Success or non-retryable error
        }

        if (attempt == maxRetries - 1) return response // Exhausted retries

        val jitter = Random.nextLong(0, currentDelay / 4)
        val waitMs = min(currentDelay + jitter, maxDelayMs)

        delay(waitMs)
        currentDelay = min(currentDelay * 2, maxDelayMs)
    }

    return block() // Final attempt
}
```

**Key parameters:**

| Parameter | Recommended Value | Description |
|---|---|---|
| `maxRetries` | `5` | Maximum number of retry attempts before giving up |
| `initialDelayMs` | `500 ms` | Wait time before the first retry |
| `maxDelayMs` | `30 000 ms` | Upper bound on the wait time between retries |
| Jitter | Up to 25% of current delay | Random noise added to prevent thundering herd |

<Tip>
  For `429 Too Many Requests` responses, check whether the API returns a `Retry-After` header. If it does, use that value as your wait time instead of the backoff calculation.
</Tip>

### Java example

```java
import java.util.Random;

public class RetryHelper {
    private static final int MAX_RETRIES = 5;
    private static final long INITIAL_DELAY_MS = 500L;
    private static final long MAX_DELAY_MS = 30_000L;

    public static void waitForRetry(int attempt) throws InterruptedException {
        long delay = Math.min(INITIAL_DELAY_MS * (1L << attempt), MAX_DELAY_MS);
        long jitter = new Random().nextLong(delay / 4);
        Thread.sleep(delay + jitter);
    }

    public static boolean shouldRetry(int statusCode) {
        return statusCode == 429 || (statusCode >= 500 && statusCode <= 599);
    }
}
```
