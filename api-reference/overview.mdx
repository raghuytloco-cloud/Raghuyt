The Stan API gives you programmatic access to the Stan platform — India's mobile social network for gamers. All resources are served over HTTPS from a single base URL, respond with JSON, and require a valid Bearer token on every request. This page walks you through the foundational concepts you need before making your first call.

## Base URL

Every request targets the same origin. The current version prefix is `/api/v1/`.

```text
https://api.getstan.app/api/v1
```

<Note>
  The Stan API fully supports HTTP/2. Clients that negotiate HTTP/2 (e.g. `okhttp/4.12.0` with the `--http2` flag) benefit from multiplexing and reduced latency. HTTP/1.1 clients work too — no special handling is required on your end.
</Note>

## Minimal Request Template

Every call to the Stan API must include a Bearer token **and** the full set of platform-identity headers. The example below shows the minimal shape of an authenticated request.

<CodeGroup>

```bash cURL
curl --http2 -X GET \
  -H "Host: api.getstan.app" \
  -H "accept: application/json, text/plain, */*" \
  -H "gaid: 9bbb0f02-5c2c-4a3b-8083-1fa26eeaedf4" \
  -H "appversion: 253" \
  -H "stallionversion: 0" \
  -H "platform: android" \
  -H "sid: 1784387335639-45684" \
  -H "ts: 1784387581188" \
  -H "appinstanceid: 76238ef97361a56425efb91beb0fdd57" \
  -H "installreferrer: utm_source=google-play&utm_medium=organic" \
  -H "countrycode: in" \
  -H "fingerprintid: 1784387336407.dba6a7" \
  -H "authorization: Bearer <YOUR_TOKEN>" \
  -H "accept-encoding: gzip" \
  -H "user-agent: okhttp/4.12.0" \
  "https://api.getstan.app/api/v1/users?username=ItzDakshbro1"
```

```http HTTP

Host: api.getstan.app
accept: application/json, text/plain, */*
gaid: 9bbb0f02-5c2c-4a3b-8083-1fa26eeaedf4
appversion: 253
stallionversion: 0
platform: android
sid: 1784387335639-45684
ts: 1784387581188
```

</CodeGroup>

<Note>
  Every request must include all 14 headers shown above. The `sid` and `ts` values are client-generated session and timestamp identifiers; `fingerprintid` is a device fingerprint. Replace the sample values with values appropriate for your client session.
</Note>

## Authentication

All endpoints require a valid Bearer JWT in the `Authorization` header. Requests without a token, or with an expired or malformed token, receive a `401 Unauthorized` response.

```http
authorization: Bearer <YOUR_TOKEN>
```

See the [Authentication](/api-reference/authentication) page for the full token structure, lifetime, and error details.

## Content Type

All Stan API responses are `application/json`. You should always include `accept: application/json` (or `application/json, text/plain, */*`) in your request headers to signal the expected content type.

## Error Format

All error responses follow a consistent JSON structure. You can rely on `statusCode` for programmatic branching and `message` for human-readable context.

```json
{
  "statusCode": 404,
  "message": "Cannot GET /",
  "error": "Not Found"
}
```

| Field        | Type     | Description                                      |
|--------------|----------|--------------------------------------------------|
| `statusCode` | `number` | Standard HTTP status code (e.g. `401`, `404`).   |
| `message`    | `string` | Human-readable description of what went wrong.   |
| `error`      | `string` | Short error label matching the HTTP status text. |

Confirmed status codes returned by the API:

| Code  | Meaning      | Typical Cause                                   |
|-------|--------------|-------------------------------------------------|
| `200` | OK           | Request succeeded.                              |
| `401` | Unauthorized | Missing, expired, or malformed JWT.             |
| `404` | Not Found    | The requested resource or route does not exist. |
